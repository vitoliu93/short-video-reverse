1分钟视频推理

概念零层 — 视频在计算机里是什么

视频 = 帧序列 + 音频

第一层拆解

帧 (Frame) = 一张图片。视频就是快速播放的图片翻页书（30fps = 每秒翻30页）

第二层拆解

图片 = 像素网格。每个像素 = 一个颜色值 (R,G,B)，范围 0-255

第三层拆解

音频 = 一串数字。每个数字代表某一时刻空气振动的幅度。44100Hz = 每秒采样 44100 次

类比：Canvas 的 getImageData() 拿到的 RGBA 数组 — 一帧视频就是那个东西。视频就是很多帧。

模型推理 = 函数调用

本质

模型就是一个函数：input → output。跟调 REST API 一样，区别在于逻辑不是人写的 if-else，是从数据里学出来的

参数 (Parameter)

= 模型内部的可调节数字。7B = 70亿个"旋钮"。训练 = 有人帮你调好所有旋钮。推理 = 你使用调好的旋钮

置信度 (Confidence)

模型输出不是"是/否"而是概率，比如"我 87% 确定这是切点"。你设阈值（threshold）决定信不信

类比：训练 = 刷了10万道题的人。推理 = 这个人做一道新题。模型文件 = 这个人的大脑状态。

子系统 A — 镜头切分 (Shot Detection)

解决的问题：视频里哪些时刻画面"跳"了？

镜头 (Shot) vs 场景 (Scene)

镜头 = 一次不间断的录制。按"录制"到按"停止"之间就是一个镜头。
场景 = 一组在相同地点/时间讲同一件事的镜头集合。

镜头切分容易（像素突变），场景理解很难（需要理解语义）。我们先做镜头切分。

TransNetV2 的工作原理

① 把每帧缩小到 48×27 像素（只看大致颜色布局）
② 用"滑动窗口"沿时间线比较相邻帧的差异
③ 有些窗口跳着看（膨胀卷积 = 看第1帧和第20帧的差异，捕捉慢渐变）
④ 每帧输出一个 0~1 的分数 = 这帧是切点的概率
⑤ 超过阈值 → 标记为切点

类比：你是 DJ 在监控人群情绪。短期传感器（1秒内有没有突变？）捕捉硬切。长期传感器（30秒前后对比）捕捉渐变。网络同时用多种时间尺度的"传感器"。

硬切 Hard Cut

第N帧是狗，第N+1帧是山。像素瞬间全变。

→ 容易检测：信号很响

渐变 Dissolve

画面A慢慢变透明，画面B慢慢浮现。20-30帧内渐进变化。

→ 难检测：需要"远距离"对比

子系统 B — 字幕提取 (OCR + Timing)

解决三个独立问题：WHERE + WHAT + WHEN

问题A：WHERE（在哪）

图里哪些区域有文字？

输出：坐标框 (bounding box)

≈ getBoundingClientRect()

问题B：WHAT（是什么）

框里的文字写了什么？

输出：字符串

≈ element.textContent

问题C：WHEN（什么时候）

这条字幕从几秒出现到几秒消失？

输出：时间戳对

≈ MutationObserver

传统方案 vs 端到端方案

PaddleOCR（传统流水线）

每帧独立处理：检测框 → 识别文字 → 跨帧比对追踪。
像是每 33ms 轮询 textContent 然后做 diff。

EVE（端到端）

一个模型，整段视频进去，字幕+时间戳直接出来。
有时间维度的理解，能处理淡入淡出。像 React 的声明式 vs jQuery 的命令式。

为什么字体识别特别难？

文字内容是有限的（26个字母/4000常用汉字），模型从已知集合里选。
字体风格是连续的、高维的——粗细、衬线、圆角、间距... 像从截图反推 CSS 一样难。
→ 解法：不精确分类，而是让 VLM（视觉语言模型）用自然语言描述风格。

子系统 C — BGM 识别 (Audio Fingerprint)

解决的问题：这段视频的背景音乐是什么？

概念链（从底到上）

① 声音 = 空气振动。麦克风把振动变成电信号，芯片把电信号变成数字序列

② 采样率 = 每秒记录多少次。44100 Hz = 每秒 44100 个数字快照

③ 频率 = 振动快慢。低频=鼓点(60Hz)，高频=哨子(4000Hz)。Hz = 每秒振动次数

④ 频谱 = 某一时刻的"成分表"。一杯果汁里30%草莓50%香蕉 → 一段声音里多少低频多少高频

⑤ 傅里叶变换 = "逆搅拌机"。把混合的声音拆解回各个频率成分。你不用学公式，只要知道它把时间波形变成频率柱状图

⑥ 频谱图 = 频谱沿时间排列。X轴=时间，Y轴=频率，颜色=音量强度。音乐的"热力图"

Shazam 算法（5步）

① 把音频变成频谱图
② 找频谱图上最亮的点（能量峰值 = 星座点）
③ 把相邻的峰值配对 → (频率A, 频率B, 时间差) 三元组
④ 把三元组变成哈希码 = 指纹
⑤ 拿指纹去数据库查：哪首歌在相同时间偏移处有相同的指纹？

类比：每首歌有独特的"星座图案"。即使只看到几颗星（几秒片段），只要相对位置对得上，就能认出是哪个星座（哪首歌）。

经典指纹 (Shazam)

手工规则：找峰值→配对→哈希
精确匹配："这是同一首歌吗？"
弱点：变速会失效

神经指纹

神经网络从配对样本中学习
输出一个向量（128个数字）
优势：变速/翻唱也能匹配

CLAP Score

音频和文字放进同一个空间
不问"是哪首"，问"风格像什么"
"找一首节奏欢快的电子乐"

为什么有噪声也能识别？

指纹只保留最响的峰值。噪声很少恰好在同样频率产生同样响的峰。就像山顶在云雾中依然突出——你认山看的是最高峰的相对位置，不是山脚的细节。

子系统 D — 多模态理解 (ARC-Hunyuan-Video-7B)

解决的问题：整体上这个视频在"讲什么"、"怎么讲的"、"想让你感受什么"

什么是"多模态"

模态 = 信息通道类型（文字/图像/声音）。人天然多模态：同时看画面+听声音+读字幕来理解视频。
有人说"我很好"但声音发抖 → 你综合判断他不好。多模态模型做同样的事。

Token（令牌）— 模型消化信息的最小单元

文字 → 拆成词块（"前端开发" → ["前端", "开发"]）
图像 → 切成小方块（16×16像素 = 1个 token）
音频 → 频谱图切成小矩形（一段时间×频率区间 = 1个 token）
关键洞察：一切都变成 token 后，同一个引擎（Transformer）可以统一处理

Transformer = 注意力引擎

核心机制"注意力"：输入的哪些部分应该关注哪些其他部分？
比如"它很累"中的"它"往回看 → 发现指的是"猫"而不是"垫子"。
视频理解中：第5秒说的话 × 第5秒的画面 → 跨模态关联。

类比：React 的 virtual DOM 解决"UI 哪些部分需要重新渲染"。Attention 解决"输入哪些部分需要互相感知"。

ARC-Hunyuan-Video 的特别之处

• 时间戳叠加：把 "00:15" 直接印到帧图片上 → 模型能读时间码 → 知道"什么时候"发生了什么
• 音视频同步编码器：音频和画面精确对齐，能理解"这个笑声配合的是哪个画面"
• GRPO 强化学习：生成多个回答 → 互相比较 → 学习更好的那些 → 事实精度提升带动主观理解提升

架构范式 — 单模型 vs 智能体编排 (AVI)

方案 A：一个大模型做所有事

ARC-Hunyuan-Video = 一个多面手独自演奏所有乐器。

  • ✓ 快速、简单、端到端
  • ✓ 适合短视频（< 5分钟）
  • ✗ 难以新增能力
  • ✗ 长视频吃不下
类比：jQuery 时代——一个库干所有事

方案 B：LLM 指挥 + 专家工具

AVI 范式 = 导演自己不演戏，编排各个演员上场时机。

  • ✓ 灵活、可扩展、可解释
  • ✓ 每个工具做一件事做到极致
  • ✓ 适合复杂/长视频
  • ✗ 编排复杂度高
类比：React + 状态管理 + API 层的微服务架构

KOX 的选择：两者结合。 ARC-Hunyuan-Video 作为 AVI 编排中的一个"专家工具"，负责整体叙事理解。TransNetV2、PaddleOCR、音频指纹等各自负责自己的子任务。最终由编排层组装成统一 JSON。
