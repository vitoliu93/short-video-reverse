## 0. 核心框架:BGM 不是一个模型,是一条小流水线

最容易犯的错是把「做 BGM」当成「找一个 BGM 识别模型」。实际它是三个**互相独立**的子问题,
对应 KOX `add_audio` 的不同参数:

| | 子问题 | 回答什么 | 对应 add_audio 参数 | 性质 |
|-|-|-|-|-|
| **A** | BGM 分离 | 把背景乐从人声/音效里抠出来 | (前置,不直接产参数) | 地基,必做 |
| **B** | BGM 识别 | 这是什么曲 / 最像库里哪条 | `audio_url` | 核心,有岔路 |
| **C** | BGM 用法刻画 | 音量曲线、进/出时机、卡不卡点 | `volume` / `start` / `end` | 确定性 DSP |

**关键认知:喂给 eval 的硬指标大半在 C,不在 B。** 只盯着「识别曲子」会漏掉真正可量化的编辑信号。

---

## 1. 逐块选型

### A. 分离 —— Demucs(HTDemucs)

**为什么是前置必做项:** 抖音音轨是「旁白人声 + 背景乐 + 音效」的混合。不分离直接处理会两头坏:

- 算音量曲线 → 旁白的大音量污染 BGM 包络,C 失真
- 做音频 embedding → 向量混进人声特征,B 的检索全乱

所以分离是 **B 和 C 的共同前提**。Demucs 是音源分离事实标准,取其 music/伴奏 stem。

### B. 识别 —— 本质岔路:指纹 vs 语义检索

两条路的差别不在工具优劣,而在**回答的是不同的问题**:

| | B1 音频指纹 | B2 CLAP 语义检索 |
|-|-|-|
| 工具 | SeekTune / neural-music-fp | LAION-CLAP(或 MERT) |
| 原理 | 频谱峰值压成哈希,精确比对 | 音频+文字映射到同一向量空间 |
| 回答 | 「是不是**同一段录音**」 | 「**听感/风格最像**哪条」 |
| 两首同 mood 不同曲 | 指纹完全不同 | 向量很近 |
| 前置依赖 | **必须有匹配用的参考曲库** | 同样需要库,但做的是近邻而非精确 |

**决策:选 B2,砍掉 B1。** 不是拍脑袋——**两种 eval 框架都指向 B2**:

1. **Gold Case 要可重放**(拆成参数能用 KOX 重渲染):`audio_url` 必须指向 `kox-statics/bgm`
   里**真实存在**的条目。抖音原曲基本不在我们库里,即便偶然在,版权可用性也不保证 → 指纹
   大概率返回「无匹配」。要的是「库里最像的那条」→ B2。
2. **Gold Case 只做统计基准**(不重放,只统计高质量视频 BGM 的风格分布/用法):连具体
   audio_url 都不需要,只要 CLAP 的**风格标签** + C 的 DSP 指标 → 还是 B2。

> 一个选型同时扛住两种未来,是它正确的强信号。

**为什么 CLAP 而非别的:** CLAP 把音频和文字塞进同一向量空间,于是一个模型同时给两种能力:
> - 文字→音频:输入「欢快电子」零样本召回 → 自动打风格标签
> - 音频→音频:两段各算向量,余弦相似度 = 听感接近度 → 库内替换检索

这正好覆盖 B 的两个需求(打标签 + 找替换),不用拼两个模型。

### C. 用法刻画 —— 纯 DSP(librosa)

零模型、确定性最高、风险最低:

- RMS 能量 → 音量包络曲线、BGM 进入/退出时间点
- beat tracking / tempo → 是否卡点、与镜头切点对不对齐

这是 eval 最硬、最可量化的指标来源,且一定能跑出来。

## 2. 关于 kox-statics/bgm 库(B2 的替换曲库)

`tos://kox-statics/bgm/` 已按功能预分类,正是 B2 的理想检索库:

| 前缀 | 条数 | 含义 |
|-|-|-|
| `lyrics_bgm/` | ~167 | 带人声/歌词 |
| `normal_bgm/` | ~441 | 常规背景乐 |
| `rhythm_bgm/` | ~416 | 卡点/节奏型 |

共约 1025 条 / 4.53GB。规模小到 CLAP 离线建索引几分钟即可。三类预分类本身可作检索的**先验桶**
(可限定在同类内检索,或把类别当作 eval 的一维标签)。拉取用 `tos-cli` skill。

## 3. 最大不确定性 → 决定了先做哪一步

诚实评估:全流程里唯一**没把握**的是 **CLAP 在这 1025 条库上的音频→音频检索质量**。
CLAP 是通用音频模型,音乐细粒度相似度未必最强;音乐专用 embedder(**MERT**)在纯音乐检索上
可能更准,但 MERT 无文字头、做不了零样本打标签。

因此排序按 **de-risk 优先**,而不是「先做简单确定的」:

| 顺序 | 动作 | 理由 |
|-|-|-|
| **第 1 步** | 只建 CLAP 索引 + 几段 query 做 top-k 检索,**人耳听**像不像 | 用最便宜的实验戳破最大未知假设 |
| 并行 | 搭 A(Demucs)+ C(librosa) | 已知能跑,无风险,可同时推进 |
| 之后 | 汇总成 bgm 结构化输出 | 前两步验证后再铺开 |

第 1 步结果分叉:
- **行** → B2 路线确认,继续
- **不行** → audio-audio 换 MERT,文字打标签仍用 CLAP

> **M0 结果(2026-05-29):走「行」分支。** 83 条样本验证:CLAP(`laion/larger_clap_music`)
> top-5 同类命中 0.83 vs 随机 0.34,人耳认可排序,**不切 MERT**。唯一坑是 raw cosine 全挤在
> 0.93~1.0 无法解读 → 检索前**减全库质心(mean-centering)**即解决,分数拉到 0.3~1.0 且排序不变。
> 另发现曲库同一首跨类重复(category 是软标签)。详见 [spec §7](./BGM-反解-spec.md#7-m0-验证结果2026-05-29)。

---

## 4. 一句话总结

**Demucs 分离打地基 → CLAP 做「风格标签 + 库内替换检索」(砍掉指纹)→ librosa DSP 出音量/时机硬指标**;
落地先用一个 CLAP 检索小实验验证最大不确定性,再铺开。

指纹路线(neural-music-fp)只在未来确认「有一批 gold 视频确实用了我们库里的原曲」时才作为精度补丁回收。
