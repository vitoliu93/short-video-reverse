## What went well (fortify)

- **先验证最危险的假设，再动工。** F0 没写任何管线，先用合成退化集量化「闭集字形比对能不能从 ~1300 款里把真字体排进 top-5」。combo 退化下 NCC top-5=0.99（随机 0.034）——这一步就决定了整个项目走不走得通。值得固化成习惯：研究型任务的第一个 milestone 永远是"把会让项目归零的假设单独拉出来证伪"。
- **重构问题定义本身（最大的智力杠杆）。** 把"开集字体分类（难、要训练、要标注）"重述成"我们手里有全部 TTF → 渲染再比对（不训练）"。一旦这么看，字重/字号这些属性也"白送"——从匹配到的字体的干净渲染里读，而不是从退化像素里量。少写一堆代码的来源是想清楚，不是写得快。
- **诚实的负结果，并且回头改了旧结论。** DINOv2 是带着"语义形状特征也许对渲染器差异更鲁棒"的假设上的，实测在低分辨率/退化下显著差于 NCC，于是不采纳，并回头把 §11 里"想提精度上 DINOv2"那句划掉、指向 §12 的证伪。没有让一个听起来高级但错的建议留在文档里。
- **合成-带标签 当 eval 骨架。** 真实抖音没 ground-truth，精度只能靠合成退化量化；真实数据只能用"投票一致性/视觉合理性"这类代理指标判断。早早认清"真实数据上能验证什么、不能验证什么"，省掉了在无标签数据上空追准确率的弯路。
- **重活丢后台。** 两条真实视频 re-extract 用后台任务跑，主线程不被 OCR 刷屏。

## Friction & fixes

- **`uv run --python .venv-font` 静默忽略 venv → ModuleNotFoundError。** 根因：目录里有 pyproject.toml 时 `uv run` 用它自己解析的环境，不认 `--python` 指的裸 venv。修复：直接 `.venv-font/bin/python scripts/x.py`。教训：项目带 pyproject 时别假设 `uv run` 会尊重外部 venv。
- **颜色极性 bug 在 F2 埋下、活到 F3 真实数据才被抓。** 白字+黑描边+亮底时，Otsu"少数派=墨"把黑描边当成了字色（#020202）。根因：假设了墨的明暗极性。修复：锚定背景色、取离背景远的像素当字形，再用距离变换从内部取填充色。**真正的教训在 eval 设计**：合成 GT 只测了"好走的" combo，没造对抗样本（亮底/深描边/浅字），所以这个 bug 合成集全过、真实集才露。合成 eval 要主动包含会打破隐含假设的边角，不只是难度梯度。
- **stale `.pyc` 掩盖 NameError。** 改完模块后另一个脚本 import 到旧字节码，跑出来像"过了"。教训：迭代被 import 的模块时清 `__pycache__`（或 `python -B`），别信一次"莫名通过"。
- **字重从退化像素量 → 全判 thin。** 在噪声 crop 上量笔宽不可靠。修复：改用 intrinsic_weight 在匹配字体的干净渲染上量。本质同上——闭集框架已把干净字形白送了，却先绕远路去量脏的。
- **ffmpeg exit 254 被 `-loglevel error` 吞掉。** 根因只是输出目录不存在。教训：写文件前 `mkdir -p`；开发期别把 ffmpeg 日志压太狠，省得错误无声。

## Corrections for the user

- **"自主跑到决策点/硬阻塞/完成"这条指令非常好用，值得复用。** 它让 agent 一口气推完 F0–F3 + 两个实验，没有为每步往返确认。对边界清晰的研究任务，这是最佳协作模式。
- **真实 eval 样本只给了 2 条。** 对字体识别，多给几条字体/背景差异大的样本，会更早压出边角 bug（颜色极性那个 bug 多两条对抗样本就能在 F3 早期暴露，而不是靠肉眼看出 #020202 不对）。下次研究型任务，eval 样本"多样性"比"数量"更重要——明确告诉 agent 你想覆盖哪些极端。
- **后续需求是"完成"之后陆续追加的**（先像素位置，再颜色修复+DINOv2+复盘）。这些都合理，但若在 F3 收尾那条消息里一次性列全想要的后续，能作为一个 phase 一起规划，而不是 done 之后再开口。
